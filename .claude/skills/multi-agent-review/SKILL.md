# multi-agent-review

PR 디프를 시야가 서로 독립인 3명의 자식 에이전트가 종합 리뷰하고, 4번째 에이전트(Orchestrator)가 결과를 합산해 신뢰도 가중 보고를 만드는 스킬.

## 언제 사용 / 사용 X

**사용**: 무거운 PR을 머지하기 전 게이트. 사용자 manual-test 직전 단계.

**사용 X**: 변경 디프 50줄 미만의 가벼운 PR (오타 수정, 스타일 정리, 단일 함수 리팩터 등). 시야 독립성 합산의 비용이 효용을 넘는다.

## 시야 독립성 모델

3명에게 **동일 프롬프트, 동일 작업(종합 리뷰)** 을 준다. 각자 독립 컨텍스트(이전 대화·부모 세션 정보 X). 같은 코드를 본 3개의 독립 시야가 일치하는 곳은 신뢰도 높고, 갈리는 곳은 추가 점검 대상이다.

"A는 SPEC, B는 보안, C는 품질" 식 **관점 분리는 사용 X** — 영역 간 갭이 생긴다. 한 명만 보는 영역은 그 한 명이 놓치면 끝이다. 시야 독립성 모델은 모두가 모든 것을 본다.

## 선행 조건

- PR 브랜치에 변경이 있는 상태 (push 여부와 무관 — 로컬 디프로도 동작)
- 일반 모드 또는 plan 모드 어디서든 호출 가능

## 절차

### 1. 디프 수집

`git diff origin/main...HEAD` (로컬 브랜치) 또는 `gh pr diff <num>` (이미 push된 PR). 결과를 다음 단계의 프롬프트에 그대로 박는다.

### 2. Reviewer 프롬프트 준비

아래 [Reviewer 프롬프트 템플릿](#reviewer-프롬프트-템플릿)을 그대로 사용. `<여기에 ... 붙여넣기>` 자리에 1단계 디프와 저장소 절대경로를 채운다.

### 3. Reviewer 1/2/3 병렬 호출

Task tool(`subagent_type: general-purpose`)로 자식 에이전트 3명 호출. **단일 메시지에서 3 호출을 묶어 병렬 실행**. 셋에게 정확히 같은 프롬프트를 전달.

### 4. 3개 보고 수집

각 Reviewer가 단일 메시지로 보고를 반환. 셋 다 끝날 때까지 기다림.

### 5. Orchestrator 호출

Task tool(`subagent_type: general-purpose`)로 4번째 자식 에이전트 호출. [Orchestrator 프롬프트 템플릿](#orchestrator-프롬프트-템플릿)에 3개 보고 본문을 그대로 박아 전달.

### 6. Orchestrator 단일 보고를 사용자에게 출력

Orchestrator가 반환한 단일 보고를 그대로 사용자에게 노출. 부모 세션이 추가로 가공·요약하지 않는다 (해석이 들어가면 시야 독립성이 깨진다).

## Reviewer 프롬프트 템플릿

```text
당신은 이 PR을 종합 리뷰하는 독립 리뷰어입니다.

## 컨텍스트
- 변경 디프:
<여기에 git diff origin/main...HEAD 본문 붙여넣기>
- 저장소 루트: <절대경로>
- 권위 있는 문서: SPEC.md, USER_JOURNEY.md, CLAUDE.md (모두 저장소 루트)

## 당신이 봐야 할 모든 것
- SPEC.md 충실도 (문구·동작 1:1 매칭)
- USER_JOURNEY 시나리오 영향
- CLAUDE.md 규약 준수
- 보안 (입력 검증, SQL injection, secret 노출)
- 정합성 (race, 트랜잭션, FK)
- 엣지 케이스 (빈 입력, 경계값, 타임존)
- 코드 품질 (중복, 가독성, 명명)

## 보고 형식
각 finding을 다음 카테고리 중 하나로 분류:
- blocker: 사용자 가시 동선 깨짐 또는 데이터 손상
- major: 정상 동선은 OK지만 엣지에서 깨짐 또는 보안 약점
- minor: 동작은 OK지만 코드 품질·일관성 이슈
- nit: 취향 차이 / 미세 개선

각 항목에 다음 명시:
- 파일:라인
- 어느 룰/시나리오와 어긋나는지
- 해결책 제안 (선택)

다른 리뷰어와 협의하지 마세요. 당신의 독립 시야가 본질입니다. 한국어로 보고.
```

## Orchestrator 프롬프트 템플릿

```text
당신은 3개의 독립 리뷰 보고를 합산하는 Orchestrator입니다.

## 입력
<Reviewer 1 보고 본문>
---
<Reviewer 2 보고 본문>
---
<Reviewer 3 보고 본문>

## 합산 룰 (신뢰도 가중)
- 3명 모두 지적 → high confidence (원래 카테고리 그대로 유지)
- 2명 지적 → review 필요 (한 단계 낮춤):
  * blocker → major
  * major → minor
  * minor → nit
  * nit → nit (그대로 — 더 낮출 카테고리 없음)
- 1명만 지적 → low confidence (별도 섹션, 카테고리는 원래대로 유지 — 격하 X — "참고만" 명시)

합의된 OK도 한 줄 요약 (몇 명이 OK 명시했는지).

## 보고 형식

## 신뢰도별 정리
- [high blocker] ... — Reviewer 1·2·3
- [review major] ... — Reviewer 1·2
- [low nit] ... — Reviewer 3
- [OK] N건 합의 (본문 생략)

## 게이트 판정
high blocker N건 → PR open 진행 / 보류

## 제약
- 합산만 한다. 자체 판단으로 보고에 없는 결함을 추가하지 않는다.
- 카테고리 격하 룰을 임의로 어기지 않는다.
- 한국어 보고.
```

## 게이트 룰

high confidence blocker **0건**이어야 PR open 진행. 1건 이상이면 멈추고 사용자에게 그대로 보고. 격하된 review/low 항목은 사용자 판단으로 처리(픽스 / 후속 PR로 분리 / 무시).

## 규칙

- Reviewer 3명에게 **동일 프롬프트** (관점 분리 X — 시야 독립성 모델의 본질)
- 자식 에이전트는 **부모 세션 정보 X** — Task tool 호출 시 부모의 이전 대화·합의·기존 발견 사항을 프롬프트에 포함시키지 않는다
- Orchestrator는 **합산만** 한다 — 자체 판단으로 보고에 없는 결함을 추가하지 않는다
- 모든 보고는 **한국어**
- 변경 디프 **50줄 미만**이면 본 스킬 생략 권장 (비용 > 효용)
